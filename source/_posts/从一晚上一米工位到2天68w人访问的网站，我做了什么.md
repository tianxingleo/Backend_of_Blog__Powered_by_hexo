截止到 2026 年 4 月 14 日 16:00，ACGTI 的成果：

站点累计浏览量超过 400w，访问人数超过 117w，GitHub 仓库收获 330 Stars、22 Fork。

对一个从热点中快速起意、以纯前端方式搭建、几天内完成上线和迭代的娱乐型小项目来说，这个传播速度远远超出了我一开始的预期。

故事要从2026年4月9号说起

那一天，我在B站刷到了一个关注的博主（就是之前和ChatGPT-4o谈恋爱的那位）发的视频，展示了她做的SBTI。那一两天，朋友圈、小红书都被这个小网站刷屏了。

我抱着看看这项目长什么样的心态去找了开源仓库，发现这居然是一个两三千行的单HTML文件。第一次看到的时候挺震撼——这么高热度的网站，居然是面向ChatGPT App vibe出来的。同时我也觉得这个项目的上限可以继续提升：单HTML文件决定了它难以维护，很多东西拓展起来不太方便，不够工程化。那如果用Vue3 rebuild一下，是不是就能极大拔高它的上限？

我当时就觉得这个热度应该会有大量的MBTI类网站复苏，先在GitHub上找了一下有没ACG相关的MBTI测试，感觉好像都不是我所想的那样，既然如此，不如自己做一个玩玩，正好不想复习高数（www结果高数考试也不算理想）。



那一个晚上，我与 Agents 做了什么。

## 技术：怎么在一天内从零到上线

技术选型上，当时唯一的目标就是尽可能快地完成开发，所以选择了自己相对熟悉的框架和语言：Vue3 + TypeScript。做的是纯前端网站，选择部署在 Cloudflare（这个决策非常正确，如果部署在自己的VPS上，流量早就把服务器打爆了，访问速度也绝对保证不了）。不过这也决定了这个项目的上限很低——既不能做数据分析，白白浪费了百万级的数据样本，也没有太多技术深度。

网页的 UI 复刻方面，首先想到的当然是 Gemini 3.1 Pro。前几天看到 GLM 5.1 宣传的网页风格复刻水平也非常高，但实际使用之后感觉不够理想（尽管比之前的 4.7 好太多）——还是没有原本设计的神韵，依然喜欢用自己的那套 AI 风味设计，拟合程度不够。更离谱的是高峰期任务跑一半直接显示”模型使用人数太多”就停掉了，那我那些未提交的不三不四的代码文件怎么办？最终还是选择了 Gemini 3.1 Pro，先在 App 的 Canvas 中跑出基于 React 的不错效果，再交给 Copilot 或 Codex 转成 Vue3 版本。这套流程虽然绕了一点，但效果比直接生成 Vue 代码好很多。

## 内容：怎么让题目不像模板问卷

题目设计是我绞尽脑汁也出不了太多好题的部分，而且”梗”味要么不足要么过多。让 DeepSeek R1 帮我改了改口吻，变成中文互联网风格，但这样的话语又太”糙”了。此外，出的题目也存在表述上的逻辑漏洞——比如一道"同意/反对"的题目里同时糅合了两种不同的前置条件或动机，像"因为A所以清理桌面图标"这种模糊表述，用户选"同意"和"反对"都说不通。昨天有两个 Issue 都是关于这类问题的，其中一个甚至写了1000多字，把每道题的逻辑漏洞逐条列了出来。依据 Issue 修改之后，题目明了很多，非常感谢那位提出 Issue 的朋友。后续加入了 i18n 多语言支持，有海峡对岸的网友提交了 PR，说我的网站在那边也有些热度，其中有些用词不太符合 zh-TW 的语言习惯，帮忙做了修改。

说到底，MBTI 测试题的核心体验不在于技术实现有多复杂，而在于题目能不能让用户会心一笑、觉得”这说的就是我”。模板式的问卷没人愿意做完，只有真正带”梗”的、贴合圈层文化的题目，才能让用户自发截图分享。这次的经验是：AI 可以帮你出题的骨架，但题目的灵魂还是要靠圈层语感和社区反馈来打磨。

## 美术：怎么让角色图形成统一视觉风格

调研的时候我看了一下 SBTI 的网页，风格上我觉得也不是我喜欢的——一是死板的一个框套一个框的 AI 设计不够好看，二是和原版 MBTI 的设计差异太大，缺了点 MBTI 的”风味”。

角色插画可能是整个项目里工作量最大的部分。我最先想到的是用 Nano Banana Pro，试了一下，效果略显生硬——我需要的图片风格是颜色简单、抽象、由几何体组成的插画，但 Nano Banana Pro 总是加入不少与风格不符的细节。它可能在画写实风格或者带文字的科研图时表现不错，但”做减法”的水平很一般。后来我换成了 ChatGPT 里的生图工具，效果意外地好，虽然和 MBTI 原版、SBTI 的插图风格还是有些区别，但相似度已经很高了。而且 SBTI 的插画多少有些丑化的感觉，确实不太好看，用现在这组插画反而不赖。

关键的一点是，所有角色图必须保持统一的视觉语言——同样的色彩饱和度、同样的几何抽象程度、同样的构图比例。如果风格参差不齐，用户一眼就能感受到”拼凑感”，整个测试的沉浸感就碎了。



## 推广：从100人小群到百万粉直播间

对于网页推广，这是一段极为离奇的经历。

周五晚上做得七七八八，先给舍友玩了一下；因为设计的角色有一半来自 BanG Dream，所以随手转发到了学校100人的邦邦小群，让大家帮忙测试。发完我就去乐队排练了——对，排练。没有任何推广计划，没有发帖，没有投流，甚至连 GitHub 链接都没挂。（这个时候错过了 Trae 的活动，如果当时拿这个选题参赛，说不定能拿奖，生活费可以++。）

排练结束回到宿舍，打开手机一看——有同学在 Q 群 @ 我，说永雏塔菲的直播间正在播我的网站。

我愣了两秒，然后光速把 Taffy 加入了角色库，第一次放了 SC。

180w+ 粉丝的 UP 主，在几万人观看的直播间里，玩着我做的网站。而我做的全部"推广"，就是把链接丢到了一个100人的 QQ 群。

从 100 人的邦邦群，到百万级虚拟主播的万人直播间，中间什么都没做。当时看 Cloudflare 的记录是 43.9w 人（IP）访问——确确实实吓了一跳，无心插柳，竟然有了 40w+ 的使用者。（哎，那个时候既没有想到把浏览量转化成 GitHub Stars，也没有想到加 Google Ad，还是学生思维害了我。）

后来我也发了小红书、小黑盒，也做了B站直播切片，但浏览量基本为零，也没有因此提高点击率（甚至远不如别人自发转发的效果好）。周日审了PR、关了Issue、做了不少Commit，解决了不少隐形Bug。周日每小时只有两万多访客，相比周六少了很多（不过这个数字本身也十分吓人了），截止到周日晚，已经有68w唯一访问者。



## 流量变现：三点后验总结

对于流量变现，很遗憾——完全没有变现。但复盘之后，核心失误可以归纳为三点：

**1. 低估了传播速度。** 在这个圈子里，Q群的传播力度远高于常见的流媒体视频平台。我没有预见到一条转发可以从100人的小群一夜之间跑到百万粉丝的直播间，所以根本没有为"突然爆了"这种场景做任何准备。

**2. 没有提前设计导流路径。** 仓库信息被放在了页面很小的角落，第二天的流量只有第一天的一半左右。后来在显眼位置加上 GitHub 链接，一天就有了约 150 Stars（发稿时 330 Stars）。如果一开始就设计好导流路径，Star 数量应该不止 500。

**3. 没有把流量转化为可持续资产。** 400w+ 的访问量如果加上 Google Ad，理论上可以收入 1w+ RMB。但"资产"不仅仅是广告收益——GitHub Stars、用户订阅、邮件列表、社媒关注、后续项目关注度，这些都是可以把一次性流量沉淀为长期价值的手段。当时一个都没有做。


---

这件事让我第一次真正理解了——“传播”和”产品”不是一回事，但它们会互相放大。

好的产品自己会说话，但如果没有抓住传播的窗口期，再好的产品也只是沉默的数据。Taffy的一场直播带来了40w+的访问，这是传播的力量；而网友自发提交Issue、PR帮助改进题目和多语言支持，这是产品的力量。可惜当时的我，既没有为传播做好准备（没有引流、没有变现方案），也没有为产品迭代预留足够的架构空间（纯前端决定了上限）。

这次错过了很多——错过了变现的窗口，错过了引流的最佳时机，错过了把一次性流量沉淀为长期社区的机会。但也正因如此，下一次当流量再来敲门的时候，我知道该怎么做。

半年前做了个校园跑脚本也拿到了不少热度，这次的 ACGTI 热度更高、传播更广。互联网感觉是玩明白了一半——但"一半"就够了，因为另一半，下次就会了。

不过话说回来，虽然没有什么物质上的收获，但以后是不是可以出去说自己独立运营了一个网站，并在四天内有400w+访问量？嘻嘻。
